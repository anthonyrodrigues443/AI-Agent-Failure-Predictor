# Dataset — Agent-Run Telemetry (literature-calibrated simulation)

## Why simulated, not "downloaded"

The task is **per-run failure prediction from agent telemetry** — one row per agent run,
features like context utilization, tool-chain depth, retry counts, and a binary
`failure` label. As of June 2026 there is **no large public tabular dataset of this
shape.** The closest real resources are *trace-localization* benchmarks — small,
human-annotated sets built to find *where* in a trace an error occurred, not to predict
run outcome from aggregate telemetry:

| Real dataset | Size | What it is | Why it can't train a 20k-row tabular predictor |
|---|---|---|---|
| TRAIL (Patronus, 2505.08638) | 148 traces / 841 errors | OpenTelemetry spans from GAIA + SWE-Bench, 20+ error taxonomy | Span-level error localization, n=148 — too small, not run-level features |
| Who&When | 127 trajectories | Multi-agent failure attribution | Attribution, not telemetry→outcome |
| MAST | 200+ traces (1,600 analyzed) | Failure-mode taxonomy | Qualitative taxonomy, no tabular features |
| TracerTraj-2.5K | ~2,000 pairs | Trajectory-error pairs | Error pairs, mixes synthetic traces |

So we **simulate**, but the generator is a *causal* step-level process whose constants
are calibrated to published failure statistics — failure **emerges** from latent
dynamics (tool-cascade corruption, context pressure, retry loops, difficulty mismatch),
it is never assigned from a single observed feature. This is documented honestly as a
limitation; the modeling lessons (which signals predict failure, why the industry rule
fails) transfer to real telemetry.

## Calibration sources (what the constants are anchored to)

- **MAST failure taxonomy** (1,600+ traces): Specification 41.8% / Coordination 36.9% /
  Verification 21.3% — i.e. context saturation is *not* the dominant cause; most failures
  are tool/coordination driven and happen *before* context fills.
- **Production failure mix (2024–25):** ~31% of failures are tool misuse / wrong tool args.
- **Context pressure:** context-retention accuracy drops 15–30% in sessions beyond ~10 turns
  → the simulator ramps per-step error probability once `context_pct > 0.7`.
- **Cascading corruption:** a bad tool call at step N poisons context for N+1…end → modeled
  as a `cascade` term that raises downstream error probability.
- **Datadog State of AI Engineering (2026):** rate-limit/error spans, retry amplification,
  cost/latency tails → modeled as retries and token growth.
- **Observability "12 signals":** context saturation, tail latency, tool errors, retrieval
  recall, streaming gaps → mapped to the observed feature set.
